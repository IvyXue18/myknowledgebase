# Obsidian 里的图片怎么自动上传？我把自己的图床配置记录一下

最近我想把 Obsidian 的图床配置整理成一篇教程，结果发生了一件很符合知识管理现状的事：配置还在正常工作，当时写过的草稿和价格对比却找不到了🤣。

插件已经装在仓库里，PicGo 也开着本地服务，过去上传的图片已经变成腾讯云 COS 链接。唯独“当时为什么这么选、每一步怎么配置”，没有留下能直接复用的版本。

索性重新整理一次。

这篇不会把所有图床都讲一遍。我会先解释这套东西为什么需要三个组件，再按我现在实际在用的链路，把 `Obsidian + Image Auto Upload Plugin + PicGo + 腾讯云 COS` 从头串起来。

## 01 为什么 Obsidian 还需要图床？

只在 Obsidian 里写笔记时，本地图片其实完全够用。截图粘贴进去，图片存在仓库的附件目录里，笔记通过 `![[图片名.png]]` 引用它，不联网也能看。

问题通常发生在内容要离开 Obsidian 的时候。

比如把 Markdown 发到博客、同步到 GitHub、交给 AI 处理，或者复制到另一个发布工具。本地图片路径只有自己的电脑认识，换一个环境，正文还在，图片就丢了。

图床做的事很简单：**把图片存到一个能通过网址访问的位置，再把本地引用替换成标准 Markdown 图片链接。**

```markdown
![图片说明](https://你的域名或云存储地址/图片.png)
```

这样，图片不再依赖某一台电脑，也不再依赖 Obsidian 自己的语法。

当然，它不是每个人都必须配置。如果你的笔记永远只在本地使用，本地附件更省心；如果你经常写教程、发博客或跨工具处理 Markdown，图床才开始变得值得。

## 02 先看懂这条链路

我一开始也会觉得：不就是传一张图片吗，为什么要装一个 Obsidian 插件，又要装 PicGo，还要开腾讯云？

把职责拆开就好理解了：

> Obsidian 负责写作和粘贴图片  
> Image Auto Upload Plugin 负责拦截图片并发起上传  
> PicGo 负责对接不同图床  
> 腾讯云 COS 负责真正存储图片并返回链接

实际过程是这样的：

1. 我在 Obsidian 里粘贴一张截图。
2. 插件把图片发送到 PicGo 的本地接口。
3. PicGo 使用配置好的密钥，把图片上传到腾讯云 COS。
4. COS 返回一条公网图片地址。
5. 插件把这条地址写回当前笔记。

我当前仓库使用的本地接口是：

```text
http://127.0.0.1:36677/upload
```

这里的 `127.0.0.1` 就是自己的电脑，`36677` 是 PicGo 默认使用的服务端口。**这条链路真正容易出错的地方，不在 Obsidian，而在三个组件有没有用同一个端口、同一套图床配置。**

> [配图 1：Obsidian → 插件 → PicGo → 腾讯云 COS 的流程图]

## 03 先选图床：不要只看“免费”

PicGo 本身不是图床，它更像一个统一上传器。腾讯云 COS、阿里云 OSS、七牛云、又拍云、GitHub、SM.MS 都可以接在后面。

如果只是临时试验，免费方案看起来很香。但图床是内容基础设施，我现在更看重三件事：链接能不能长期控制、服务是否稳定、费用能不能预估。

下面这张表按个人写作场景整理，价格核对时间是 **2026 年 7 月 22 日**。云厂商价格会变，正式开通时仍要以控制台为准。

| 方案 | 当前可核实的主要费用 / 免费额度 | 优点 | 需要注意 | 我的判断 |
| --- | --- | --- | --- | --- |
| 腾讯云 COS | 成都标准存储 `0.099 元/GB/月`；中国大陆外网下行 `0.5 元/GB`；新个人用户有 50GB 标准存储 6 个月免费额度，但不抵扣流量 | PicGo 原生支持，国内访问稳定，当前配置已经跑通 | 存储、请求和下行流量分开计费；要防盗链和设告警 | **这篇教程的主方案** |
| 阿里云 OSS | 官方计费示例中的标准存储为 `0.12 元/GB/月`；外网流出另计 | 云产品成熟，PicGo 原生支持 | 价格受地域、冗余类型和流量方式影响，资源包容易买错 | 已经在阿里云生态里可以选 |
| 七牛云 Kodo | 官方仍提供免费额度；计费示例包含 10GB 标准存储和 10GB CDN 回源流量免费额度，但官方同时提示目录价格会调整 | 小规模使用的免费额度友好，图床用户多 | 自定义域名、CDN 与备案等配置更容易劝退新手；旧示例价格不能直接当成今天报价 | 愿意多配置一步，可以比较 |
| 又拍云 | 标准存储在“日存储量大于日 CDN 流量”时为 `0.129 元/GB/30天`；请求、上传和 CDN 回源流量免费 | 存储和 CDN 结合紧，图片业务经验多 | 费用规则和 COS/OSS 不完全同口径，不能只比一列存储单价 | 有网站分发需求时再看 |

这张表里最容易被忽略的是：**个人图床通常不是存储贵，而是图片被大量访问后的下行流量贵。**

按我现在使用的腾讯云成都地域做一个假设：存 5GB 图片，一个月产生 10GB 外网下行，暂不计少量请求费，大约是：

```text
5GB × 0.099 元 + 10GB × 0.5 元 = 5.495 元/月
```

这不是固定套餐，只是帮你理解费用结构。图片只在自己的笔记里偶尔访问，流量可能很少；图片被博客或别的网站大量引用，流量才会成为大头。

## 04 第一步：在腾讯云 COS 创建存储桶

进入腾讯云控制台，搜索“对象存储 COS”，新建一个存储桶。

建议这样选：

- 名称：自己能看懂即可，例如 `obsidian-images`。最终完整名称会带 APPID。
- 地域：优先选离主要访问者近的地域；选定后不方便迁移。我的配置使用成都。
- 存储类型：标准存储。
- 访问权限：如果要让 Markdown 里的图片直接显示，需要允许图片被公开读取；写入权限仍然只交给自己的密钥。
- 版本控制：个人图床初期可以不开，避免误操作产生额外存储。

创建完成后，先记下三个值：

1. 完整存储桶名称；
2. APPID；
3. 所在地域，例如 `ap-chengdu`。

这些值一会儿都要填进 PicGo。

> [配图 2：创建 COS 存储桶，标出名称、地域、访问权限]

## 05 第二步：单独创建一把上传密钥

这一步很容易为了省事直接用主账号密钥，但我不建议这样做。

腾讯云官方也明确建议：不要使用主账号直接操作云资源，应该创建子账号，并按最小权限原则授权。图床密钥只需要操作这个存储桶，不应该顺手拿到整个腾讯云账号的权限。

我的建议是：

1. 在访问管理 CAM 中创建一个只用于 PicGo 的子用户；
2. 开启编程访问，生成 SecretId 和 SecretKey；
3. 权限只覆盖刚才创建的 COS 存储桶；
4. 密钥只在创建时保存，不要发到聊天记录、截图或 Git 仓库里；
5. 如果密钥曾经出现在公开位置，直接禁用并重新生成。

> [待补充：腾讯云 CAM 针对单个 Bucket 的最小读写策略截图或策略 JSON。发布前不要用“全资源管理”权限代替。]

**所有教程截图都要把 SecretId、SecretKey、APPID 和真实存储桶名称打码。**

> [配图 3：子用户 API 密钥页面，敏感信息完整打码]

## 06 第三步：在 PicGo 里配置腾讯云 COS

安装并打开 PicGo，在“图床设置”里选择“腾讯云 COS”，新建一份配置。

需要填写的字段大致如下：

| PicGo 字段 | 填什么 |
| --- | --- |
| SecretId | 子用户 API 密钥的 SecretId |
| SecretKey | 子用户 API 密钥的 SecretKey |
| Bucket | COS 的完整存储桶名称 |
| AppId | 腾讯云账号的 APPID |
| 存储区域 | 例如 `ap-chengdu`，必须和 Bucket 地域一致 |
| 存储路径 | 可选，例如 `images/` 或 `obsidian/` |
| 自定义域名 | 没有绑定域名时先留空 |
| 版本 | COS v5 |

保存后，把这份配置设为默认图床，先在 PicGo 里单独拖一张测试图片。

如果 PicGo 能上传，并在相册里返回一个 `https://...myqcloud.com/...` 链接，说明云端这半段已经通了。先把这里测通，再去折腾 Obsidian，会少很多无效排查。

> [配图 4：PicGo 腾讯云 COS 配置页，密钥与真实 Bucket 打码]

接着打开 PicGo 设置，找到 Server 服务：

- 监听地址：`127.0.0.1`
- 端口：`36677`
- 开启 Server

> [配图 5：PicGo Server 设置，标出 36677 端口]

## 07 第四步：连接 Obsidian

在 Obsidian 的社区插件市场搜索并安装 `Image Auto Upload Plugin`。启用后进入插件设置：

| 设置项 | 建议值 |
| --- | --- |
| 自动上传粘贴图片 | 开启 |
| Uploader | PicGo |
| PicGo Server | `http://127.0.0.1:36677/upload` |
| 上传后替换图片链接 | 开启 |
| 上传后删除本地图片 | **第一次配置时先关闭** |

我自己的仓库已经开启了上传后删除本地图片，但新配置时不建议一上来就这么做。先保留本地副本，连续测试几次，确认云端链接稳定、Obsidian 重启后仍能显示，再决定要不要自动删除。

> [配图 6：Obsidian 插件设置页，标出自动上传和 Server 地址]

现在回到一篇测试笔记，粘贴一张截图。成功后，正文里应该出现类似这样的内容：

```markdown
![image.png](https://examplebucket.cos.ap-chengdu.myqcloud.com/20260722120000.png)
```

可以再做三个检查：

1. 关闭 PicGo 后再粘贴，应该会失败——这能确认 Obsidian 确实依赖本地 Server；
2. 重新打开 PicGo，确认端口仍是 36677；
3. 把图片链接复制到无痕窗口，确认公网读取正常。

## 08 已有笔记里的本地图片怎么办？

这个插件不只处理新粘贴的图片，也支持批量上传当前笔记里的本地图片。

打开 Obsidian 命令面板，搜索：

```text
upload all images
```

执行前先做一份 Git 提交或仓库备份。批量操作会修改 Markdown 中的图片链接，一旦中途配置错误，单张排查会比第一次配置更麻烦。

我会先拿一篇测试笔记跑通，而不是直接对整个知识库动手。

## 09 最常见的几个坑

### PicGo 能上传，Obsidian 不能

优先检查端口。PicGo Server 和插件地址必须一致；一个是 `36677`，另一个写成别的端口，就不会通。

### 上传成功，但图片显示 403

通常是存储桶读取权限或防盗链规则的问题。先确认对象允许读取，再检查 Referer 白名单是否拦截了当前发布平台。

### 粘贴后一直没有反应

确认 PicGo 正在运行、Server 已开启。插件调用的是本机接口，PicGo 退出后，这条链路就断了。

### 图片突然产生很多费用

检查有没有被盗链。腾讯云 COS 支持 Referer 黑白名单，也可以配置外网下行流量告警。白名单适合来源固定的网站；如果文章会被多个平台读取，规则要谨慎，否则自己也可能看不到图。

至少做两件事：打开费用告警，定期看外网下行流量。不要等到账单异常才第一次进入监控页面。

### 换图床后旧链接怎么办

这也是为什么我不太建议把免费第三方图床当长期基础设施。正文里已经写入的都是旧域名，换服务不是改一个设置，而是迁移图片并批量替换历史链接。

**图床真正的成本，不只是每 GB 多少钱，也包括未来迁移一次要改多少内容。**

## 10 最后，我为什么留下这套方案

这套配置不算最轻。它有四个组件，要管理云账号、密钥、存储桶和本地服务。

但对我来说，它解决的是一个很具体的问题：我可以继续在 Obsidian 里顺手截图、粘贴、写作，同时让图片变成其他工具也认识的标准链接。

工具当然可以换。PicGo 后面可以接别的对象存储，Obsidian 插件也可以换，真正值得留下的是这条稳定的写作链路：**图片在粘贴的那一刻就完成上传，而不是等到发布时再补作业。**

这篇先把我目前在用的腾讯云 COS 版本记录下来。后面如果真的换成自定义域名、CDN，或者把图片迁移到别的对象存储，我再把迁移过程补成下一篇。
