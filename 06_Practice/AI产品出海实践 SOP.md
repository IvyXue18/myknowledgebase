# 一、需求阶段

## 工具篇

1. Similarweb：查看网站流量，流量来源渠道，地域分布，相似网站……
2. Semrush：查看网站搜索流量，搜索关键词分布，每个关键词流量大小，看词的排名，按KD值，国家地区，CPC，搜索量等，判断关键词优化难度。
3. Google trends：看搜索量趋势，是否平稳，是否是季节词，还是新词，可以多个关键词对比看流量。
4. trends radar，trends bar两款插件：https://trends.google.com/trends
5. Ahrefs：其他的可能用不到，但可以看关键词优化难度。主要关注难度和外链数据：https://ahrefs.com/keyword-difficulty
6. 可以看网站不同时期版本的迭代：https://web.archive.org/web
7. 域名查询：https://query.domains/
8. ICO 网站，不同尺寸的图标：https://favicon.io/favicon-converter/

### 一些概念

**PV：**页面访问量

**UV：**独立访客数量

**PV/UV：**页面平均访问量，衡量用户粘性，和平均页面访问时长一样

**搜索流量/有机搜索/自然搜索**：从搜索引擎过来的，分为自然搜索和付费搜索。

**直接流量**：

- 直接输入URL或者点击收藏夹和插件进去的。
- 如果google判断不出来是哪来的流量，就会归为直接流量：例如桌面软件，社群链接跳转过来的等。
- 加了noreferral标签的，比如很多网盟，广告跳转的链接等

**引荐流量**：从其他非社交网站点击引流过来的，类似于外链。

- 主站的马甲站，子工具站
- 多语言站点：后缀不一样
- affiliate代理商网站
- 付费做的高质量的引流软文
- 支付通道的服务采用商

让我逐个解释图片中红框中显示的SEO（搜索引擎优化）术语：

**反向链接：**指其他网站链接到你的网站的外部链接。这些链接对SEO很重要，因为它们像是其他网站对你网站的"投票"或背书。

**锚链接**：也称为锚文本链接，是超链接中可见的可点击文本部分。比如在"点击这里查看详情"中，"点击这里"就是锚文本。

**引荐域名**：指向你网站的链接所在的域名。例如，如果网站A链接到你的网站，那么网站A的域名就是引荐域名。

**编入索引页面**：已被搜索引擎爬虫发现并收录到搜索引擎数据库中的网页。

**出站域名**：你的网站链接指向的外部网站域名。这是与引荐域名相反的概念。

### DNS 解析（用 Vercel 需要）

进入 cf 的 dns 配置

把 vercel 这边的DNS records 写入 cf，如果 cf 已经有一个，需要删除，比如 A 记录，只能留一个。发现 vercel 只让配置 A 记录，就不用管其他的了。

vercel 的 DNS（第二个 tab）不用写入 cf。

你的域名同时指向 Cloudflare 与 Vercel，配置冲突，需要统一到一方

你在 Cloudflare 已激活并用橙云代理，但 Vercel 面板显示“Invalid Configuration”，要求把 NS 改到它的 **ns1.vercel-dns.com**** /****ns2.vercel-dns.com**，且给出 A 记录**216.198.79.1**。这造成两套权威 DNS 的竞争：Cloudflare 与 Vercel DNS只能选其一。

有两条可行路径，选一条并彻底执行。

**路径 A：保留 Cloudflare（推荐）**

- 在 Cloudflare 的 DNS 中：
  - 把 apex A 记录改为 Vercel 的目标。Vercel近期给出的 IP 是你看到的**216.198.79.1**，使用它即可，保持“已代理”。
  - 将 www 改为 CNAME 指向 bulkmedia.online 或 Vercel 提供的别名，保持“已代理”。
- 保持域名的 NS 指向 Cloudflare 的 **gigi.ns.cloudflare.com / kenneth.ns.cloudflare.com**，不要改到 Vercel。
- 在 Vercel 域名设置里选择“使用外部 DNS（Cloudflare）”，让它只做托管，不接管权威 DNS。Vercel 会检测到记录匹配后不再报错。

优点是你继续享受 Cloudflare 的缓存、WAF、规则与边缘证书；配置也更灵活。

***1.持续发布能满足搜索意图的内容（23%）***

内容为王，在2025年依然如此。但关键不在于发布多少内容，而是需要持续不断的更新，以及发布的内容是否真正解决了用户的问题。

通过semrush 或者similarweb  找到相关关键词的细分关键词，不断去拓词，每个关键词都去做一个精品内页，能够做好on-page-seo。还可以使用 Google Search Console 分析用户搜索查询，了解真实搜索意图。

***2.标题（Meta Title）包含目标关键词（14%）***

标题不仅要包含关键词，更要吸引用户点击。好的标题既要满足搜索引擎，也要满足用户需求。能够匹配上你的要做的关键词。

- 标题长度控制在50-60个字符以内，避免在搜索结果中被截断。
- 内容的单词数不能太少，需要600字以上，核心关键词的密度建议是 3% 左右。
- 网站的heading 结构清晰，能够很清晰的让Google 知道你网站是做什么的。

***3.外链（13%）***

外链依然是谷歌排名的重要因素。但现在更注重外链的质量而非数量，一条高质量外链胜过一百条垃圾外链。同时相关性也是比较重要，和你网站内容相关的外链肯定是大于无关的外链。

可以从竞品网站外链开始，使用 semrush 或者similarweb 分析竞品外链来源，自己也可以提交相应的外链。在平时做的过程中，也需要把这些外链收集起来，下次做新的网站，就可以直接去发了。

***4.搜索者参与度（12%）***

用户的行为数据直接影响搜索排名。用户在你网站上停留时间越长，浏览页面越多，对排名越有利。

- 优化页面加载速度，确保3秒内完全加载
- 使用清晰的网站导航，确保用户能在3次点击内找到需要的内容
- 通过分段标题、列表、图片和视频等多媒体形式提升内容可读性
- 在文章末尾添加相关内容推荐，增加页面浏览深度
- 设计互动元素：工具、计算器、测试等，增加用户参与度

***5.新鲜度（6%）***

搜索引擎喜欢新鲜的内容。保持网站内容的更新频率，对排名很有帮助。

- 建立内容发布计划，保持定期更新
- 及时更新过时的内容和数据
- 在现有文章中添加最新信息和案例
- 关注行业动态，第一时间发布相关内容
- 使用结构化数据标记最后修改时间

***6.移动优先（5%）***

谷歌已经实行移动优先索引，移动端的用户也是占了半壁江山，移动端体验直接影响所有设备的搜索排名。有些网站就是通过更好的移动端体验打败了一些更强大的网站。

***8.可信度（4%）***

网站的可信度影响用户是否愿意在你的网站上停留和转化。建立信任是长期经营的基础。

- 启用HTTPS加密
- 展示真实的客户评价和使用案例
- 建立专业的隐私政策和使用条款页面

***9.外链分布多样性（3%）***

外链来源的多样性比单一来源的大量外链更有价值。要从不同类型的网站获得外链。

- 从不同权重的网站获得外链：高权重门户网站、中等权重行业网站、小网站
- 多样化外链类型：文章内链接、资源页链接、目录提交、社交媒体
- 控制锚文本分布：品牌词45%、完全匹配30%、部分匹配25%
- 地理分布多样化：不同国家和地区的网站
- 获得不同类型网站的外链：博客、新闻站、目录站、论坛等

***10.页面速度（2%）***

虽然权重不高，但页面速度直接影响用户体验，进而影响参与度评分。

- 优化图片：使用WebP格式，设置合适的尺寸
- 使用CDN加速静态资源加载
- 压缩CSS和JavaScript文件
- 选择高性能的托管服务器
- 定期使用Google PageSpeed Insights检测和优化

***11.站内链接（1%）***

合理的内链结构帮助搜索引擎更好地理解网站内容层次，同时提升用户浏览深度。

- 建立清晰的网站层级结构：首页→分类页→内容页
- 在内容中自然地添加指向相关页面的链接
- 使用描述性的锚文本，避免"点击这里"等无意义词汇
- 定期检查和修复死链接
- 在文章结尾推荐相关内容，形成内链网络

***12.技术优化和其他细节（1%）***

这包括各种技术细节和优化措施，虽然权重不高，但积少成多。

- 优化URL结构，使用描述性的URL

# 自问自答

###  shipany新建一个组件放在 components 还是ui？

用户询问 `@components/` 和 `@ui/` 目录的关系，以及如何决定新组件的放置位置。

当你需要创建一个新组件时，你可以遵循以下原则来决定它的放置位置：

1. **判断组件是否是 UI 基础组件？**
  - **如果是**：这个组件是 UI 的最小、可重用部分吗？它是否不包含任何特定的业务逻辑？它是否可以在应用程序的多个不同部分被直接复用，而不需要知道其父组件的上下文？
    - **示例**：你创建了一个新的 `CustomDatePicker` 组件，它只是一个日期选择器，没有关联任何数据提交逻辑，可以在任何地方选择日期。
    - **放置位置**：`**src/components/ui/**`
2. **判断组件是否包含业务逻辑或更复杂的组合？**
  - **如果不是 UI 基础组件**：这个组件是否组合了多个 `ui` 组件，并包含了特定的业务逻辑、数据获取或页面布局？它是否更专注于解决某个特定功能或展示某个特定区域？
